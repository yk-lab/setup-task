## 1. 背景 / Background

`task`（go-task）を GitHub Actions で利用する際のデファクトは [`arduino/setup-task`](https://github.com/arduino/setup-task) だが、運用上の課題が顕在化している。

### 1.1 観測された具体的課題

1. **保守の停滞**
   最新リリースは **v2.0.0（2024-02-06）** で以降更新がない。go-task 本体は活発に更新され続けている（最新 v3.51.1）一方、セットアップ側が追従していない。

2. **GitHub Actions ランタイム廃止リスク**
   `arduino/setup-task` の `action.yml` は `using: "node20"`。GitHub は Node 20 ランタイムを廃止予定で、**2026-06-16 から Node 24 が強制**、**2026-09-16 に Node 20 をランナーから削除**する。薄い JS アクションのため強制 Node 24 でも動く見込みだが、保守停滞と相まって中長期のリスクとなる。

3. **未認証 GitHub API レート制限による一過性失敗**（実際に踏んだ事象）
   リリース解決のために叩く GitHub API が**未認証**だと、共有ランナーの IP 単位レート制限（60 req/h）に当たり、API が HTML エラーページ（通称 "Unicorn" ページ）を返す。アクションはこれをバイナリ／JSON として処理しようとして失敗する。`repo-token` を渡せば緩和できるが、既定では未設定のため踏みやすい。

4. **チェックサム検証の不在**
   ダウンロードしたバイナリの完全性検証（SHA256 照合）を行わない。go-task は `task_checksums.txt` を公開しているにもかかわらず活用されておらず、**供給網（supply-chain）セキュリティ**の観点で弱い。

5. **ツールキャッシュ未活用**
   `actions/tool-cache` によるランナー間／ジョブ間キャッシュがなく、毎回ダウンロードが走る。

### 1.2 課題の出どころ

本企画は、実プロジェクト（pnpm monorepo の CI）で `arduino/setup-task@v2` が "Unicorn" 404 を掴んで Lint ジョブが落ちた事象、および Node 20 廃止警告の検知を契機とする。再実行で復旧したが、**根本原因（未認証レート制限 / 保守停滞）が残る**ため恒久対策として新アクションを企画する。

## 2. 目的・ゴール / Goals

> **「task を、速く・壊れにくく・検証可能に CI へ導入できる」セットアップアクションを提供する。**

| # | ゴール | 達成基準（KPI 候補） |
|---|---|---|
| G1 | 一過性 DL 失敗をほぼ根絶する | 既定で `GITHUB_TOKEN` 認証＋リトライ。CI 上の transient 失敗率 < 0.1% |
| G2 | ランタイム廃止に強い構成にする | `using: node24`。第三者ランタイム依存を最小化 |
| G3 | 供給網セキュリティを担保する | 既定でチェックサム検証 ON。検証失敗時は確実に fail |
| G4 | 高速にする | tool-cache ヒット時のセットアップ p95 < 2s |
| G5 | 移行を容易にする | `arduino/setup-task` と互換の input 名（`version` / `repo-token`）でドロップイン置換可能 |
| G6 | 主要プラットフォームを網羅する | Linux / macOS / Windows × amd64 / arm64 を CI matrix で常時 green |

## 3. 提供価値・差別化 / Value Proposition

| 価値 | 本アクション | arduino/setup-task@v2 |
|---|---|---|

| 保守 | アクティブ（自動更新運用） | 停滞（2024-02 以降なし） |

## 4. スコープ / Scope

### 4.1 In Scope（やること）
- `task` バイナリの **バージョン解決**（exact / semver range / latest）
- **プラットフォーム判定**（OS × arch → リリース資産名）
- GitHub Releases からの **ダウンロード**（既定で認証付き）＋**リトライ**
- **チェックサム検証**（SHA256）
- アーカイブ展開（`.tar.gz` / `.zip`）
- **tool-cache** への登録・復元
- `PATH` への追加
- **outputs** 提供（`version` / `task-path` / `cache-hit`）

### 4.2 Out of Scope（やらないこと）
- Taskfile（タスク）の**実行**そのもの（本アクションは「導入」まで）
- `task` 設定ファイルの生成・編集
- `.deb` / `.rpm` / `.apk` 等の**OS パッケージ形式インストール**（archive 方式に限定）
- go-task 以外のツールのセットアップ

## 6. 競合分析 / Competitive Analysis

| 手段 | 長所 | 短所 | 本企画との関係 |
|---|---|---|---|
| `arduino/setup-task@v2` | デファクト・実績 | Node20 廃止・未認証レート制限・checksum なし・保守停滞 | **置換対象** |
| `@go-task/cli`（npm） | lockfile でピン・既存 `pnpm install` に同梱可 | `postinstall` で結局 GH releases から DL・build-script 承認が必要 | 補完／fallback ソース候補 |
| 公式 `install.sh`（curl \| sh） | 手軽 | **remote script を shell 直結＝供給網リスク**・checksum/キャッシュなし | 非採用 |
| `mise` / `asdf` | 多ツール統合管理 | task 単体には過剰・別途セットアップが必要 | 別レイヤー |
| ワークフローで直接 `run` | 依存ゼロ | 各リポで再実装・保守分散 | 本アクションが代替 |

> **結論**: 既存に「Node24 × 認証既定 × checksum × cache × 移行容易」を同時に満たすものがない。ここに新規アクションの存在価値がある。

## 8. リスクと対策 / Risks & Mitigations

| リスク | 影響 | 対策 |
|---|---|---|
| GitHub Releases 側の障害・レート制限 | DL 失敗 | 認証既定化＋指数バックオフ・リトライ＋**fallback ソース**（npm tarball 等、将来） |
| Marketplace 名称衝突（`setup-task`） | 公開不可 | owner 名前空間（`yk-lab/setup-task`）＋表示名調整 |
| go-task の資産命名変更 | 解決破綻 | 命名規則を設定化・テストで早期検知・version ピン |
| 保守継続性 | 再び陳腐化 | 最小依存・Dependabot・リリース自動化で省力運用 |
| Node ランタイム再廃止 | 将来の作業 | ランタイム差し替えが容易な薄い実装に保つ |

## 9. 技術スタック（想定）/ Tech Stack

- **言語**: TypeScript
- **ランタイム**: `node24`（`action.yml` の `runs.using`）
- **主要ライブラリ**: `@actions/core` / `@actions/tool-cache` / `@actions/http-client`（or `@actions/github` for API）/ `@actions/exec`
- **バンドル**: `@vercel/ncc` で `dist/index.js` に単一ファイル化（依存を同梱しコミット）
- **テスト**: Vitest（ユニット）＋ GitHub Actions の self-test ワークフロー（matrix 結合テスト）
- **品質**: ESLint / Prettier / typecheck、CI 必須化
