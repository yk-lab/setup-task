## 1. 概要・目的 / Overview

本アクションは GitHub Actions ワークフローのステップとして、指定された `task`（go-task）バイナリを **解決 → 取得 → 検証 → 展開 → キャッシュ → PATH 追加** し、後続ステップで `task` コマンドを利用可能にする。`arduino/setup-task` のドロップイン代替を目指す。

```yaml
# 利用イメージ / Usage
- uses: yk-lab/setup-task@v1
  with:
    version: 3.x                     # 任意。省略時は既定（§FR-1）
    repo-token: ${{ secrets.GITHUB_TOKEN }}  # 任意。省略時も自動利用を試行（§FR-3）
- run: task lint
```

---

## 2. 用語 / Glossary

| 用語 | 定義 |
|---|---|
| task / go-task | Taskfile を実行するタスクランナー。本アクションが導入対象とするバイナリ |
| 資産 (asset) | GitHub Release に添付されたファイル（例: `task_linux_amd64.tar.gz`） |
| tool-cache | `@actions/tool-cache` が管理するランナー上のツールキャッシュ領域 |
| semver range | `3.x` / `^3.50` 等のバージョン範囲指定 |
| ランナー (runner) | ワークフローを実行する GitHub Actions のマシン |

---

## 3. 対象範囲・前提 / Assumptions

- **対応ランナー OS**: `ubuntu-*` / `macos-*` / `windows-*`（GitHub-hosted・self-hosted 双方を想定）
- **ランタイム**: アクションは `node24` で動作する（§CON-1）
- **取得元**: 既定で `github.com/go-task/task` の Releases（§FR-3）
- **配布形式**: archive 方式（`.tar.gz` / `.zip`）のみ対象（§4 Out of Scope, 企画書）

---

## 4. 機能要求 / Functional Requirements

### FR-1 バージョン解決
- `version` input を受け付ける。次を解決できること:
  - **exact**: `3.51.1` / `v3.51.1`（`v` 接頭辞は許容・正規化）
  - **semver range**: `3.x` / `^3.50` / `>=3.40 <3.52`
  - **latest**: `latest`（最新安定版）
- 省略時の既定は **企画オープン論点（§10-2）で確定**。確定までの暫定既定は `latest` とし、ドキュメントで「安定ピン推奨」を明示する。
- range / latest の解決は GitHub Releases の一覧から最新の合致版を選ぶ。プレリリースは既定で除外する。

### FR-2 プラットフォーム判定
- 実行ランナーの OS / CPU アーキテクチャを判定し、対応する資産名を導出する。
- 命名規則（go-task の実資産に準拠）:
  - アーカイブ: `task_<os>_<arch>.tar.gz`（`linux` / `darwin` / `freebsd`）、`task_windows_<arch>.zip`
  - `<os>` ∈ { `linux`, `darwin`, `windows`, `freebsd` }
  - `<arch>` ∈ { `386`, `amd64`, `arm`, `arm64`, `riscv64` }（OS により提供範囲は異なる、§9 参照）
- Node の `process.platform` / `process.arch`（`x64→amd64`, `arm64→arm64` 等）をマッピングする。
- `architecture` input（任意）で arch を明示上書きできること。

### FR-3 ダウンロードと認証
- 解決した資産を Releases からダウンロードする。
- **既定で認証付き**: `repo-token` が与えられればそれを、無ければ環境の `GITHUB_TOKEN` を自動利用して GitHub API / asset 取得を認証する。これによりレート制限由来の transient 失敗（HTML エラーページ混入）を回避する。
- 認証トークンが一切利用不可の場合でも動作は試みるが、レート制限リスクを `warning` で告知する。
- レスポンスが期待 content-type（バイナリ／JSON）でない（HTML エラーページ等）場合は**ダウンロード失敗として扱い**、リトライ対象とする（§FR-4）。

### FR-4 リトライ／バックオフ
- ネットワーク／一時的 HTTP エラー（429 / 5xx / content-type 不整合）に対し、**指数バックオフ**でリトライする。
- リトライ回数・初期待機はデフォルト値を持ち、必要なら input で調整可能（既定: 3 回程度）。
- 恒久エラー（404 = 該当版なし、checksum 不一致）はリトライせず即時 fail する。

### FR-5 チェックサム検証
- リリースの `task_checksums.txt` を取得し、ダウンロードした資産の **SHA256** を照合する。
- **既定で検証 ON**。不一致は即時 fail（§FR-4 のリトライ対象外）。
- `task_checksums.txt` が取得できない版・状況では、`skip-checksum`（任意 input・既定 false）が true の場合のみ警告付きで継続。既定では fail とする。

### FR-6 展開 / 設置
- `.tar.gz` / `.zip` を展開し、`task` 実行ファイル（Windows は `task.exe`）を取り出す。
- 実行権限を付与する（POSIX）。

### FR-7 ツールキャッシュ
- `@actions/tool-cache` に **キー（ツール名 + 解決済み version + os + arch）** で登録・復元する。
- キャッシュヒット時はダウンロード／検証をスキップする。

### FR-8 PATH 追加
- 設置先ディレクトリを `core.addPath()` で `PATH` に追加し、後続ステップから `task` を解決可能にする。

### FR-9 出力 / Outputs
- 次の outputs を提供する:
  - `version`: 実際にインストールした解決済みバージョン（例: `3.51.1`）
  - `task-path`: `task` 実行ファイルの絶対パス
  - `cache-hit`: tool-cache ヒット有無（`true` / `false`）

### FR-10 入力 / Inputs
- §6 の inputs を受け付ける。`arduino/setup-task` 互換のため `version` / `repo-token` の名称・意味を一致させる（§NFR-4）。

### FR-11 フォールバックソース（将来 / 任意）
- GitHub Releases が継続失敗する場合に備え、**代替取得元**（例: npm の `@go-task/cli` tarball 同梱バイナリ）からの取得を将来サポートできる設計とする。v1 必須ではない（企画 §10-3 で判断）。

## 5. 非機能要求 / Non-Functional Requirements

### NFR-1 セキュリティ
- チェックサム検証を既定 ON とする（FR-5）。
- トークンはログ・サマリに出力しない（`core.setSecret` 相当の取り扱い）。
- 取得 URL は go-task 公式リリースに固定し、リダイレクト先を検証する。
- 依存は最小限・ピン留めし、`dist/` バンドルを監査可能にする。

### NFR-2 信頼性
- transient エラー耐性（FR-3 / FR-4）。同一入力で**冪等**に同じ結果を返す。
- 失敗時は理由が一意に判別できるメッセージで終了する（§8）。

### NFR-3 パフォーマンス
- tool-cache ヒット時のセットアップ **p95 < 2 秒**（目標）。
- 不要な API 呼び出しを避ける（exact 指定時は一覧取得を省略可能に）。

### NFR-4 互換性 / 移行性
- `arduino/setup-task` からの移行が `uses:` 行の差し替えのみで済むよう、`version` / `repo-token` の input 名・意味を互換に保つ。
- 既存ユーザーの代表的ワークフローがそのまま通ることを移行ガイドで保証する。

### NFR-5 可観測性
- `core.info` / `core.debug` で解決版・取得元・cache 有無・検証結果をログ出力する。
- ジョブサマリ（`core.summary`）に導入結果を記録する（任意）。

### NFR-6 保守性
- TypeScript + 単一責務モジュール構成（解決 / 取得 / 検証 / 展開 / キャッシュ）。
- ユニットテスト＋ self-test matrix（§10）で回帰を防ぐ。
- Dependabot・リリース自動化で更新を省力化する。

## 6. インターフェース仕様 / Interface（`action.yml`）

### 6.1 Inputs

| 名前 | 必須 | 既定 | 説明 |
|---|---|---|---|
| `version` | 任意 | `latest`（暫定／要確定） | 導入する task のバージョン。exact / semver range / `latest`（FR-1） |
| `repo-token` | 任意 | `${{ github.token }}` を試行 | GitHub API/asset 取得の認証トークン（FR-3） |
| `architecture` | 任意 | ランナー自動判定 | CPU アーキテクチャ上書き（FR-2） |
| `check-latest` | 任意 | `false` | range 指定時に常に最新合致版を再解決するか |
| `skip-checksum` | 任意 | `false` | チェックサム未取得時のみ継続を許可（FR-5、非推奨） |

### 6.2 Outputs

| 名前 | 説明 |
|---|---|
| `version` | 解決・インストールされたバージョン（FR-9） |
| `task-path` | `task` 実行ファイルの絶対パス |
| `cache-hit` | tool-cache ヒット有無（`true`/`false`） |

### 6.3 Runs

```yaml
runs:
  using: "node24"      # CON-1
  main: "dist/index.js"
```

## 7. 動作シーケンス / Behavior Flow

```
1. inputs 読込・正規化（version / arch / token）
2. version 解決（exact はそのまま / range・latest は Releases 一覧から選定）   … FR-1, FR-3(認証)
3. tool-cache 照合（key: task + version + os + arch）
   ├─ HIT → 9 へ（cache-hit=true）                                            … FR-7
   └─ MISS ↓
4. 資産名導出（OS/arch → task_<os>_<arch>.{tar.gz|zip}）                       … FR-2
5. ダウンロード（認証付き・リトライ/バックオフ）                               … FR-3, FR-4
6. checksums 取得 → SHA256 照合（不一致は即 fail）                            … FR-5
7. 展開 → 実行権限付与                                                        … FR-6
8. tool-cache へ登録                                                          … FR-7
9. PATH 追加・outputs 設定（version / task-path / cache-hit）                  … FR-8, FR-9
```

## 8. エラーハンドリング / Error Handling

| 事象 | 分類 | 挙動 |
|---|---|---|
| 該当バージョンなし（404 / range 不一致） | 恒久 | 即 fail。利用可能版の候補を提示 |
| ダウンロード一時失敗（429/5xx/HTML 混入） | 一時 | バックオフでリトライ（FR-4）、上限超過で fail |
| checksum 不一致 | 恒久・セキュリティ | 即 fail。取得物を破棄しメッセージ明示 |
| checksums 取得不可 | 条件付き | `skip-checksum=true` のみ警告継続、既定は fail |
| 未対応 OS/arch | 恒久 | 即 fail。対応表（§9）を案内 |
| トークン不使用 | 警告 | 続行するがレート制限リスクを `warning` |

- すべての fail は `core.setFailed()` で非ゼロ終了し、原因が一意に分かる文言を出す。

## 9. プラットフォーム対応表 / Platform Matrix

go-task の実リリース資産（v3.51.1 時点）に基づく。

| OS (`<os>`) | 形式 | 対応 arch |
|---|---|---|
| `linux` | `.tar.gz` | `386` / `amd64` / `arm` / `arm64` / `riscv64` |
| `darwin` | `.tar.gz` | `amd64` / `arm64` |
| `windows` | `.zip` | `386` / `amd64` / `arm64` |
| `freebsd` | `.tar.gz` | `386` / `amd64` / `arm` / `arm64` |

- チェックサムは全資産分が単一の `task_checksums.txt` に集約される（FR-5 はこれを参照）。
- `.deb` / `.rpm` / `.apk` は対象外（企画 §4.2）。

## 10. テスト要求 / Test Requirements

### 10.1 ユニットテスト
- version 解決（exact / range / latest / `v` 接頭辞 / プレリリース除外）
- OS/arch → 資産名マッピング（全 §9 組合せ）
- content-type 判定（HTML エラーページを失敗と判定）
- checksum 照合（正常／改ざん検出）
- リトライ／バックオフ（一時失敗→成功、上限超過→fail）

### 10.2 結合（self-test）ワークフロー matrix
- OS: `ubuntu-latest` / `macos-latest` / `windows-latest`（可能なら arm ランナーも）
- version 指定: exact / `3.x` / `latest`
- 各組合せで `task --version` が解決版と一致することを検証
- cache hit / miss 双方の経路を検証

### 10.3 セキュリティ・回帰
- checksum 改ざんを注入して必ず fail することを確認
- トークン未指定時の警告継続を確認

## 12. 制約・非対象 / Constraints

| ID | 制約 |
|---|---|
| CON-1 | アクションランタイムは `node24`。Node 20 は採用しない |
| CON-2 | 取得元は go-task 公式リリースに限定（任意の URL 指定は許可しない） |
| CON-3 | remote script の shell 直結（`curl \| sh`）は採用しない（供給網セキュリティ方針） |
| CON-4 | archive 方式（`.tar.gz`/`.zip`）のみ。OS パッケージ形式は非対象 |
